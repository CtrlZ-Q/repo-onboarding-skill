# repo-onboarding Skill

快速理解当前代码库，生成一份面向开发者和 AI Agent 的项目入门报告。

## 目标

在进入陌生仓库、接手新任务、修 bug 或准备改代码之前，先回答：

1. 这个项目是什么？
2. 用什么语言、框架和包管理器？
3. 怎么安装、启动、测试和构建？
4. 哪些目录和入口文件最重要？
5. 核心执行流程是什么？
6. 有哪些项目约定、风险点和不要乱动的地方？
7. Claude 后续在这个仓库里应该如何安全工作？

## 何时使用

用户出现以下意图时：

- "帮我读一下这个项目"
- "这个仓库是做什么的"
- "先熟悉一下项目"
- "生成项目说明 / onboarding 文档"
- 初次进入陌生仓库准备开发
- 大型改动前需要摸清项目结构
- 准备生成或更新项目级 `CLAUDE.md`

## 深度模式

根据用户请求推断深度：

- **Quick**：快速定位，只看根目录、技术栈、常用命令和核心目录。用户说"快速看一下"时使用。
- **Standard**（默认）：完整报告，包含架构、执行流、测试、风险和下一步建议。
- **Deep**：Standard 基础上，深入追踪核心流程、依赖边界、CI/部署、持久化、测试质量和维护风险。

## 证据纪律

结论必须基于仓库证据。

**强证据**：
- README 和文档
- 包管理配置和 lockfile
- 构建、测试、lint 配置
- CI workflow 文件
- Docker / compose / 部署文件
- `.env.example` 等环境变量示例
- 入口文件和路由表
- 测试代码

**弱证据**：
- 仅凭文件名猜测
- 未在本地确认的框架约定
- 从依赖名推测业务领域
- README 中过时的说明

使用弱证据时必须标注"推测"。命令如果只是从配置推断而未实际运行，标注"未验证"。

## 执行流程

### 1. 先运行自动扫描脚本

在项目根目录运行：

```bash
node "${CLAUDE_SKILL_DIR}/scripts/scan-project.mjs"
```

如果无法确定 skill 目录，直接读取 `scripts/scan-project.mjs` 的内容来理解扫描逻辑，或用 Glob/Grep 工具手动完成同样的扫描。

脚本输出结构化 Markdown，包括：
- 根目录文件
- 检测到的语言和框架
- 包管理器
- 常用 scripts
- 配置文件
- 候选入口文件
- 测试相关文件
- 风险提示

### 2. 阅读关键文档

优先阅读顺序：
1. `CLAUDE.md`（项目级指令）
2. `README.md` / `README.zh-CN.md`
3. `docs/` 下的核心文档
4. 包管理 / 构建配置
5. 测试配置
6. CI workflow

**禁止读取**真实 `.env`、密钥文件或凭据文件。可以读取 `.env.example`。

### 3. 确定项目类型

从以下类别中选一个：
- application / service / library / CLI / monorepo / plugin / template / data project / documentation site / mixed

### 4. 追踪执行流（Standard 和 Deep 模式）

找到入口文件后，追踪至少一条从入口到核心逻辑的路径：

- 前端：`main.tsx` → App 组件 → 路由 → 页面 → API 调用
- 后端：`main.ts` → 服务初始化 → 路由注册 → 中间件 → 控制器 → 数据访问
- CLI：入口 → 命令注册 → 命令处理器

用具体文件和函数名描述，不要写抽象概述。

### 5. 理解核心行为（Standard 和 Deep 模式）

通过源码和测试（测试是行为文档）识别：
- 核心业务对象
- 重要工作流和状态转换
- 持久化模型
- 集成点（外部 API、数据库、消息队列）
- 服务边界

### 6. 确定常用命令

命令来源优先级：
1. 项目文档明确说明
2. `package.json scripts` / `Makefile` / `pyproject.toml` / `Cargo.toml` / `go.mod`
3. CI workflow
4. 测试框架配置推断

每条命令标注状态：
- `已验证`：实际运行成功
- `已配置`：在配置文件中找到，未运行
- `仅文档`：README 中提到但配置中未找到
- `推测`：从框架约定推断

如果 README 与 lockfile 冲突（例如 README 写 `npm run dev`，但仓库有 `pnpm-lock.yaml`），明确说明冲突和推荐选择。

### 7. 梳理目录

只总结有意义的目录，不输出完整文件树。

每个核心目录说明：
- 负责什么
- 后续改某类问题应该先看哪里
- 是否为生成代码、构建产物或第三方依赖

### 8. 标记不确定信息

不要把猜测写成事实。

错误写法：
> 这是一个电商系统。

更好的写法：
> README 未明确说明业务领域。从 `cart`、`checkout`、`orders` 等目录看，项目可能涉及电商流程，但需要人工确认。

信息冲突时列出两个来源，给出推荐，标注需要确认。

### 9. 评估质量（Deep 模式）

- 测试覆盖信号和重要缺失
- CI 覆盖情况
- 配置安全性
- 依赖风险信号
- 代码组织和边界清晰度
- 生成代码或构建产物

### 10. 大仓库策略

大仓库不要全量读取。优先看：
- 根目录文档和配置
- 工作区 / 包管理配置
- 应用/包入口文件
- 构建和 CI 配置
- 核心业务目录
- 重要流程的测试
- 部署配置

声明采样策略和未覆盖的部分。

### 11. 生成报告

使用 `templates/project-onboarding-template.md` 作为结构参考。

输出位置：
- 如果存在 `docs/`：写入 `docs/project-onboarding.md`
- 否则：写入 `PROJECT_ONBOARDING.md`

生成完成后，向用户简要汇报：
- 报告写入路径
- 项目类型
- 推荐启动/测试命令
- 发现的主要风险点

## 原则

- 不修改业务代码
- 不自动提交
- 不默认覆盖现有 `CLAUDE.md`
- 不读取 `.env`、私钥、token、证书等敏感内容
- 不猜测无法从代码或文档确认的事实
- 信息冲突时明确列出冲突来源
- 大仓库不要全量读取，优先看根配置、文档、入口和测试配置
- 输出要帮助后续 Agent 安全开发，而不是只做目录摘要

## 可选增强

如果用户要求更新 `CLAUDE.md`：

1. 先生成"建议加入 CLAUDE.md 的片段"
2. 展示给用户确认
3. 再写入或编辑 `CLAUDE.md`

建议片段应只包含项目特有、长期有效、能降低后续错误率的规则，例如：
- 使用 pnpm，不要使用 npm
- 不要手改 `src/generated/`
- 修改 API 后必须运行某个测试命令
- 数据库迁移只能新增，不能改历史迁移
